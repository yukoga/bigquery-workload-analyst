This guide outlines advanced Workload Management (WLM) architecture solutions derived from Google Cloud best practices and the BigQuery Cost Optimization Whitepaper. Use these strategies to isolate workloads, guarantee service level agreements (SLAs), and control costs.

---

## 1. Workload Isolation via Reservations

To prevent low-priority batch or ad-hoc queries from starving critical, customer-facing dashboards or interactive APIs, establish distinct capacity pools using BigQuery Reservations (Editions).

### Reservation Mapping Strategy
Create three core reservations under your administration project:

```
                  [ BigQuery Capacity Pool ]
                              |
       +----------------------+----------------------+
       |                      |                      |
[ prod-reservation ]   [ batch-reservation ]   [ adhoc-reservation ]
- Baseline: 300 slots  - Baseline: 100 slots   - Baseline: 0 slots
- Max Autoscale: 1000  - Max Autoscale: 500    - Max Autoscale: 200
- Priority: CRITICAL   - Priority: LOW         - Priority: MEDIUM
- Customer APIs / BI   - Scheduled ELT (dbt)   - Developer Sandboxes
```

1. **`prod-reservation` (Production Dashboard & API Workloads)**:
   - **Characteristics**: Low-latency, high-concurrency SLA requirements.
   - **Configuration**: Assign dedicated baseline slots (e.g., 300 slots) with flexible autoscale ceilings (e.g., up to 1000 slots).
2. **`batch-reservation` (Scheduled Batch, ETL, and ELT Pipelines)**:
   - **Characteristics**: Large data-scanning scheduled tasks (e.g., dbt runs, Airflow tasks). High throughput but flexible completion windows.
   - **Configuration**: Set a lower baseline (e.g., 100 slots) but high autoscale ceiling (e.g., 500 slots) to handle execution bursts. Disable idle slot sharing to prevent ad-hoc interference if SLAs are strict.
3. **`adhoc-reservation` (Interactive & Playground Developer Sandboxes)**:
   - **Characteristics**: Exploration, query development, and manual testing. High variance and unpredictable execution profiles.
   - **Configuration**: Set baseline to 0 slots and utilize pure autoscaling with strict slot ceilings (e.g., maximum 200 slots) to prevent runaway costs from manual mistakes.

## 2. Dynamic Query Queuing

When query demand exceeds the allocated slot capacity of a reservation, BigQuery's built-in **Query Queuing** feature must be optimized to preserve system stability and prevent failure.

### Key Tuning Recommendations
- **Interactive Concurrency**: Enable query queuing to automatically buffer excess queries instead of returning resource-exhausted errors.
- **Queue Timeout Control**: Set the maximum queue timeout (`queue_timeout_ms`) depending on the reservation's profile:
  - For `prod-reservation` (Dashboards): Set to a low limit (e.g., `120000ms` / 2 minutes) so that front-end clients fail fast and notify users rather than hanging indefinitely.
  - For `batch-reservation` (Scheduled ETL): Set to a generous limit (e.g., `14400000ms` / 4 hours) to allow batch pipelines to queue up and execute sequentially during high-load periods.

## 3. BI Engine Memory Allocation

For high-frequency, repetitive dashboard queries (such as those generated by Looker, Looker Studio, or Tableau), rely on **BigQuery BI Engine**'s in-memory acceleration.

### Implementation Checklist
- **Memory Allocation**: Reserve BI Engine memory (up to 100GB per project/region) in the main production analytical project.
- **Target Table Definition**: Explicitly configure BI Engine metadata to target active physical tables or materialized views backing active dashboards.
- **SQL Tuning**: Avoid non-accelerated SQL operators (such as complex custom JavaScript UDFs or massive multi-way outer joins) in dashboard queries to ensure BI Engine can accelerate execution in-memory, consuming **0 slots** from the reservation pool.

## 4. Runaway Cost Safeguards

Establish defensive guardrails to prevent single un-optimized queries from draining your budget or reservation capacity.

### Configuration Controls
1. **Query-Level Bytes Billed Limits**:
   - For all interactive/sandbox environments, enforce the `maximum_bytes_billed` connection property.
   - Example: Restrict developers to a maximum of 10GB (`10,000,000,000` bytes) per query. If a query exceeds this limit, BigQuery fails it *before* scanning begins.
2. **User/Project Custom Quotas**:
   - Establish daily scanning quotas at the user or project level using Google Cloud Console quotas.
   - Example limit: Maximum of 50TB of scanned data per project per day for developer sandboxes.

## 5. Physical Optimization & Materialization

To reduce raw workload size before WLM rules are applied:
- **Partitioning**: Enforce mandatory partition filters on heavy time-series tables (e.g., partition by `creation_time` or `DATE(event_timestamp)`).
- **Clustering**: Cluster partitioned tables by frequently filtered or aggregated dimensions (e.g., `user_id`, `state_code`) to prune data scans at the block level.
- **Materialized Views**: Maintain automatic, incremental materialized views for heavy aggregated metrics to prevent raw scanning of base tables during recurrent queries.
